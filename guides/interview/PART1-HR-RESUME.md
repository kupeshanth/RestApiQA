## Q9: Tell me about a challenging bug you found.

**A:**

One bug that I am particularly proud of finding was during my time at Qoria, while testing the EdTech Insight reporting module.

The feature allowed school administrators to generate usage reports for students across a selected date range. The happy-path tests all passed cleanly. However, during exploratory testing, I decided to test the scenario where the start date of the report range was set to a date that fell on a daylight saving time transition day — specifically testing across a UTC offset boundary.

What I found was that for users in certain time zones, the report was silently dropping the first hour of data from the first day of the range. The system was converting timestamps correctly in most cases, but there was a timezone offset mismatch in the database query layer that caused records in that boundary hour to be excluded. The report would show data but it was subtly incomplete — which is actually more dangerous than a crash, because users would trust the data without knowing it was wrong.

The challenge was that this was extremely hard to reproduce consistently because it depended on the test user's system timezone, the server timezone, and the specific date chosen. I documented it meticulously — the exact dates, timezone settings, the discrepancy count between expected and actual records — and I added a data comparison script to verify the record count before and after the bug fix was deployed. The developer confirmed it was a real defect and it was fixed before the feature went to production. That experience reinforced to me that the most dangerous bugs are the silent data integrity ones, and exploratory testing across boundary conditions is essential for catching them.

## Q13: Describe a conflict with a developer and how you handled it.

**A:**

I had logged a bug because when an API call failed due to a network timeout, the application displayed a generic "Something went wrong" message with no guidance for the user. From a UX and testability standpoint, I marked it as a medium-severity defect because it degraded the user experience and made it impossible for administrators to self-diagnose the issue.

The developer who had built that feature pushed back quite directly, saying it was not a bug and that the design spec did not define what the error message should say. He felt I was raising it as a defect when it was actually a missing requirement, and he wanted to reject it outright rather than fix it.

I did not escalate immediately or argue in the moment. Instead, I acknowledged his point about the spec gap — he was technically correct that the spec was silent on error message content. Then I added a comment to the JIRA ticket with a reference to the general UX guidelines our product uses, which did state that user-facing errors should provide actionable information. I also logged the user impact clearly: if a school administrator sees a blank error, they will raise a support ticket rather than retry, which increases support load unnecessarily.

I suggested we discuss it in the next sprint grooming session rather than arguing at the defect triage. When the product owner saw the JIRA comment with the UX guideline reference and the support impact reasoning, she agreed it should be fixed. The developer was professional about it — he fixed it in the next sprint. What I learned from that is that conflicts with developers are best resolved with evidence, not emotion, and that involving the right stakeholders at the right time is more effective than a direct standoff.

## Q14: Tell me about a time you handled a production issue.

**A:**

Early in my time at Qoria, there was an incident where a report export feature that had been released to production started generating corrupt PDF files for some users. The support team escalated it, and it became an all-hands-on-deck situation for the QA and dev team.

My specific role was to help isolate the reproduction pattern. The bug was not happening for everyone — it was environment-specific and user-data-specific, which made it tricky. I went through the support tickets to identify patterns in the affected users: what role they had, what data they were exporting, what browser they were using. I then set up a systematic test matrix to reproduce it in our staging environment using similar data profiles.

After about two hours of structured investigation, I found that the bug only occurred when the exported report exceeded a certain record count threshold AND the user had a specific custom column configuration applied. Below that threshold or with the default configuration, the export worked fine. This narrowed the issue to a specific code path that the developers could then target directly.

I documented the reproduction steps, created a regression test specifically for this scenario, and after the hotfix was deployed, I ran a targeted verification pass in production with a test account. I also added that scenario to our regression suite so it would never slip through again.

It was a high-pressure situation, but it reinforced for me that structured, methodical investigation is the right approach even when there is urgency — and that having a QA engineer who can isolate reproduction steps quickly is genuinely valuable in an incident.

It is a role that keeps me engaged across multiple types of thinking throughout the day, which I genuinely enjoy.

## Q16: Explain your role in Scrum ceremonies.

**A:**

I participate in all four main Scrum ceremonies and I have a distinct contribution in each.

In **Sprint Planning**, my role is to analyse the user stories being pulled into the sprint from a testability perspective. I raise questions about acceptance criteria that are ambiguous, I flag stories that have dependencies or environment requirements that could block testing, and I give rough estimates for how much testing effort each story will require. I also start thinking about which stories will need automation updates versus just manual execution.

In **Daily Standup**, I give a brief update on what I tested the previous day, what I am testing today, and whether any defects I raised are blocking progress. I keep it short and flag blockers clearly so the Scrum Master can help resolve them.

In **Sprint Review**, I contribute by demonstrating the test coverage we achieved — I show the test results, any defects found and their resolution status, and I provide a quality summary for the features that are being showcased. If a story was not fully tested due to a late code delivery, I flag it clearly so the product owner can make an informed release decision.

In **Sprint Retrospective**, I contribute feedback about process improvements in the QA workflow. For example, I once raised that we were consistently getting features to test with only two days left in the sprint because of late code handovers, and I proposed we add a "dev complete" checkpoint mid-sprint to give QA earlier visibility. That proposal was adopted in the next sprint. I see retrospectives as important quality improvement opportunities, not just team social sessions.
