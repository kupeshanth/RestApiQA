## SECTION 1 — ALLURE FUNDAMENTALS

**Q1: What is Allure Report and why would you use it instead of the default Maven Surefire report?**

**A:**

Allure Report is an open-source, interactive test reporting framework that transforms raw test result data (JSON files) into a rich HTML dashboard. It is maintained by QA Guru and integrates with TestNG, JUnit, Cucumber, RestAssured, Selenium, and Playwright.

The default Maven Surefire report (generated by `mvn surefire-report:report`) produces a basic HTML page with a table of test names and pass/fail status. It does not show:
- Which step in the test failed
- Screenshots or API request/response at the point of failure
- Grouping by business domain (epic/feature/story)
- Test execution history and trend graphs
- Severity classification of failures
- Links to JIRA or test management systems

Allure answers these questions in its dashboard:
- **Behaviours tab**: Tests grouped as Epic > Feature > Story — aligns with business requirements
- **Suites tab**: Tests grouped by class and method — technical view
- **Timeline tab**: When each test ran and how long
- **Graphs tab**: Severity breakdown, pass rate donut chart, duration graph
- **Categories tab**: Failures grouped by type (product defect, test defect, environment issue)
- **History/Trend tab**: Pass rate across the last N builds

A manager can look at the Allure dashboard and understand release health in under 30 seconds. A developer can click into a failing test and see the exact step, the request sent, the response received, and a screenshot — all in one place.

```xml
<!-- pom.xml — Allure dependencies -->
<properties>
    <allure.version>2.24.0</allure.version>
    <aspectj.version>1.9.20.1</aspectj.version>
</properties>

<dependencies>
    <dependency>
        <groupId>io.qameta.allure</groupId>
        <artifactId>allure-testng</artifactId>
        <version>${allure.version}</version>
        <scope>test</scope>
    </dependency>
    <dependency>
        <groupId>io.qameta.allure</groupId>
        <artifactId>allure-rest-assured</artifactId>
        <version>${allure.version}</version>
        <scope>test</scope>
    </dependency>
</dependencies>

<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-surefire-plugin</artifactId>
            <version>3.1.2</version>
            <configuration>
                <!-- AspectJ weaver is required for @Step to work at bytecode level -->
                <argLine>
                    -javaagent:"${settings.localRepository}/org/aspectj/aspectjweaver/${aspectj.version}/aspectjweaver-${aspectj.version}.jar"
                </argLine>
            </configuration>
        </plugin>
    </plugins>
</build>
```

Note: The AspectJ weaver is required because `@Step` works by intercepting method calls at bytecode level. Without it, `@Step` annotations are silently ignored and nothing appears in the report.

**Q2: What do @Epic, @Feature, and @Story annotations do in Allure?**

**A:**

These three annotations create a three-level hierarchy in the Allure "Behaviours" tab. The hierarchy is designed to map directly to how teams structure work in project management tools like JIRA:

```java
@Epic("Order Management")      // highest level — business epic
@Feature("Order Creation")     // mid level — feature within the epic
public class CreateOrderTest {

    @Test
    @Story("Create order with valid payload returns 201")   // test-level story
    public void createOrder_validPayload_returns201() { ... }

    @Test
    @Story("Create order with missing customerId returns 400")
    public void createOrder_missingCustomerId_returns400() { ... }
}
```

In the Allure dashboard the Behaviours tab renders:
```
Epic: Order Management
  └── Feature: Order Creation
        ├── Story: Create order with valid payload returns 201
        │     └── createOrder_validPayload_returns201 [PASSED]
        └── Story: Create order with missing customerId returns 400
              └── createOrder_missingCustomerId_returns400 [PASSED]
```

A Product Owner reading the Behaviours tab can see exactly which user stories have passing tests, without understanding any code. This is the primary value of these annotations — they bridge the gap between test execution and business requirements.

You can also add multiple labels programmatically at runtime:
```java
Allure.epic("Order Management");
Allure.feature("Order Creation");
Allure.story("Happy path — standard order");
```

**Q3: What does @Description do and how should you write a good description?**

**A:**

`@Description` adds a human-readable description of what the test verifies, displayed in the report alongside the test name. There are two forms:

```java
// Plain text description
@Test
@Description("Verify that POST /api/orders with a valid payload returns HTTP 201, " +
             "contains an orderId starting with ORD-, and persists the record to Spanner " +
             "with status=PENDING and the correct customer_id and amount.")
public void createOrder_validPayload_returns201() { ... }

// HTML description (for richer formatting)
@Test
@Description(value = "<h4>Test Objective</h4>" +
             "<p>Verify POST /api/orders happy path end-to-end:</p>" +
             "<ul><li>HTTP 201 status</li><li>orderId in response</li><li>Data in DB</li></ul>",
             useMarkdown = false)
public void createOrder_validPayload_returns201_html() { ... }
```

Write descriptions that answer:
1. What endpoint or feature is being tested?
2. What is the specific scenario (happy path, edge case, error case)?
3. What does the test verify end-to-end?

Avoid descriptions that just repeat the test method name. "Tests POST /api/orders" adds no value. "Verify that a valid POST /api/orders request persists the order in Spanner with all fields correct and returns a 201 with an orderId" tells a reviewer exactly what would break if the test fails.

**Q4: What does @Step do and how does it work with the AspectJ weaver?**

**A:**

`@Step` marks a method as a named step. Every time that method is called during a test, Allure records it in the test timeline with its name, the actual parameter values, and a pass/fail status. Steps can be nested — if a step method calls another step method, Allure shows the nested structure.

```java
@Step("Build order payload: customerId={customerId}, amount={amount}, type={type}")
private String buildOrderPayload(String customerId, double amount, String type) {
    return String.format(
        "{\"customerId\":\"%s\",\"amount\":%.2f,\"type\":\"%s\"}",
        customerId, amount, type
    );
}

@Step("Send POST /api/orders")
private Response sendCreateOrderRequest(String payload) {
    return RestAssured.given()
        .contentType("application/json")
        .body(payload)
        .post("/api/orders");
}

@Step("Verify HTTP status is {expectedStatus}")
private void verifyStatusCode(Response response, int expectedStatus) {
    assertEquals(response.getStatusCode(), expectedStatus,
        "Status mismatch. Body: " + response.getBody().asString());
}
```

The `{paramName}` placeholders in the step name are replaced with actual values at runtime. If you call `buildOrderPayload("CUST-42", 149.99, "PRIORITY")`, the report shows:

```
Step 1: Build order payload: customerId=CUST-42, amount=149.99, type=PRIORITY  [PASSED]
Step 2: Send POST /api/orders  [PASSED]
Step 3: Verify HTTP status is 201  [PASSED]
```

The AspectJ weaver is the mechanism that intercepts the method call and records it in Allure. Without `aspectjweaver` in the `-javaagent` argLine in pom.xml, `@Step` does nothing — the annotation is present but never processed.

You can also create inline steps programmatically without a dedicated method:
```java
@Test
public void createOrder_test() {
    Allure.step("Send POST /api/orders", () -> {
        Response response = given().body(payload).post("/api/orders");
        Allure.step("Verify status 201", () -> assertEquals(response.getStatusCode(), 201));
    });
}
```

**Q5: What does @Attachment do and what types can you attach?**

**A:**

`@Attachment` attaches a file to the test result in the Allure report. The method must return `byte[]` (for binary files like images) or `String` (for text content). Allure stores the attachment alongside the test result and renders it inline in the report.

```java
// Screenshot — returns byte array
@Attachment(value = "Screenshot at failure", type = "image/png", fileExtension = "png")
public byte[] attachScreenshot(WebDriver driver) {
    return ((TakesScreenshot) driver).getScreenshotAs(OutputType.BYTES);
}

// Text log
@Attachment(value = "Application log", type = "text/plain", fileExtension = "txt")
public String attachLog(String logContent) {
    return logContent;
}

// JSON body (API response)
@Attachment(value = "API Response Body", type = "application/json", fileExtension = "json")
public String attachResponseBody(String body) {
    return body;
}

// HTML content
@Attachment(value = "Page source", type = "text/html", fileExtension = "html")
public String attachPageSource(WebDriver driver) {
    return driver.getPageSource();
}
```

You can also attach content programmatically inside a test or step without a dedicated method:
```java
// Attach a string
Allure.addAttachment("Request payload", "application/json", payload, "json");

// Attach bytes (e.g. a PDF)
byte[] pdfBytes = Files.readAllBytes(Paths.get("report.pdf"));
Allure.addAttachment("Generated PDF", "application/pdf", 
    new ByteArrayInputStream(pdfBytes), "pdf");
```

**Q6: What does @Severity do and what are the severity levels?**

**A:**

`@Severity` classifies the importance of a test. The Allure dashboard can display a breakdown by severity and lets you filter to show only the most critical failures.

```java
@Test
@Severity(SeverityLevel.BLOCKER)   // blocking release if failing
public void payment_gatewayDown_blocksCheckout() { ... }

@Test
@Severity(SeverityLevel.CRITICAL)  // core business path
public void createOrder_validPayload_returns201() { ... }

@Test
@Severity(SeverityLevel.NORMAL)    // standard test (default if no annotation)
public void getOrder_withOptionalFields_returns200() { ... }

@Test
@Severity(SeverityLevel.MINOR)     // edge case, low impact
public void getOrder_withLongDescription_truncatesGracefully() { ... }

@Test
@Severity(SeverityLevel.TRIVIAL)   // cosmetic or very low value
public void apiResponse_hasCorrectContentTypeHeader() { ... }
```

In practice:
- **BLOCKER**: The release cannot ship if this fails (payment processing, login, account creation)
- **CRITICAL**: Major feature broken, high customer impact (order creation, checkout flow)
- **NORMAL**: Regression tests for existing features (most tests)
- **MINOR**: Edge cases, pagination, optional fields
- **TRIVIAL**: Header checks, optional validation messages

The Allure Graphs tab shows a severity donut chart so leadership can quickly see "3 BLOCKER failures, 12 CRITICAL failures" instead of reading a list.

**Q7: What do @Issue and @TmsLink annotations do?**

**A:**

These annotations create hyperlinks from the Allure test result to external systems:

```java
@Test
@Issue("OMS-1234")       // links to JIRA issue OMS-1234
@TmsLink("XRAY-567")     // links to Xray or Zephyr test case XRAY-567
public void createOrder_validPayload_returns201() { ... }
```

For the links to be clickable, you must configure the URL pattern in `allure.properties` (placed in `src/test/resources/`):

```properties
# allure.properties
allure.link.issue.pattern=https://jira.yourcompany.com/browse/{}
allure.link.tms.pattern=https://jira.yourcompany.com/browse/{}
```

The `{}` placeholder is replaced with the value you pass to the annotation. So `@Issue("OMS-1234")` becomes a link to `https://jira.yourcompany.com/browse/OMS-1234`.

There is also a generic `@Link` for any URL:
```java
@Link(name = "API Documentation", url = "https://docs.example.com/orders-api")
@Link(name = "Design Spec", url = "https://confluence.example.com/display/SPEC/Orders")
```

In the Allure report, each test shows these as clickable icons. A developer investigating a failure can jump directly to the JIRA issue or the test case specification without searching.

**Q8: Complete example — a test class using all Allure annotations together.**

**A:**

```java
package com.example.tests;

import io.qameta.allure.*;
import io.qameta.allure.restassured.AllureRestAssured;
import io.restassured.RestAssured;
import io.restassured.response.Response;
import org.testng.annotations.BeforeClass;
import org.testng.annotations.Test;
import static org.testng.Assert.*;

@Epic("Order Management")
@Feature("Order Creation")
public class CreateOrderTest {

    @BeforeClass
    public void setUp() {
        RestAssured.baseURI = System.getProperty("api.base.url", "https://api.example.com");
        RestAssured.filters(new AllureRestAssured());  // auto-logs all requests/responses
    }

    @Test
    @Story("Create order with valid payload returns 201")
    @Description("Send a well-formed POST /api/orders and verify: HTTP 201, orderId in response starting with ORD-, record persisted in Spanner with status=PENDING.")
    @Severity(SeverityLevel.BLOCKER)
    @Issue("OMS-100")
    @TmsLink("XRAY-200")
    public void createOrder_validPayload_returns201() {
        String payload = buildPayload("CUST-42", 99.99, "STANDARD");
        Response response  = sendCreateOrder(payload);
        verifyStatus(response, 201);
        String orderId = verifyOrderIdInResponse(response);
        verifyDbRecord(orderId);
    }

    @Test
    @Story("Create order with missing customerId returns 400")
    @Description("POST /api/orders without customerId should return 400 with a meaningful error.")
    @Severity(SeverityLevel.CRITICAL)
    @Issue("OMS-101")
    public void createOrder_missingCustomerId_returns400() {
        String payload = "{\"amount\":99.99,\"type\":\"STANDARD\"}";
        Response response = sendCreateOrder(payload);
        verifyStatus(response, 400);
        verifyErrorMessage(response, "customerId is required");
    }

    // ── Steps ──────────────────────────────────────────────────────

    @Step("Build payload: customerId={customerId}, amount={amount}, type={type}")
    private String buildPayload(String customerId, double amount, String type) {
        return String.format("{\"customerId\":\"%s\",\"amount\":%.2f,\"type\":\"%s\"}",
            customerId, amount, type);
    }

    @Step("POST /api/orders")
    private Response sendCreateOrder(String payload) {
        return RestAssured.given()
            .contentType("application/json")
            .header("Authorization", "Bearer " + System.getenv("TEST_AUTH_TOKEN"))
            .body(payload)
            .post("/api/orders");
    }

    @Step("Verify HTTP status is {expectedStatus}")
    private void verifyStatus(Response response, int expectedStatus) {
        assertEquals(response.getStatusCode(), expectedStatus,
            "Status mismatch. Body: " + response.getBody().asString());
    }

    @Step("Verify orderId is present in response body")
    private String verifyOrderIdInResponse(Response response) {
        String orderId = response.jsonPath().getString("orderId");
        assertNotNull(orderId, "orderId should not be null");
        assertTrue(orderId.startsWith("ORD-"), "orderId should start with ORD-");
        return orderId;
    }

    @Step("Verify order {orderId} exists in database")
    private void verifyDbRecord(String orderId) {
        // DB verification via Spanner or JDBC
        Allure.addAttachment("DB check", "Order " + orderId + " found with status=PENDING");
    }

    @Step("Verify error message contains: {expected}")
    private void verifyErrorMessage(Response response, String expected) {
        String actual = response.jsonPath().getString("message");
        assertTrue(actual != null && actual.contains(expected),
            "Expected error message '" + expected + "' but got: " + actual);
    }
}
```

**Q9: How do you add the AllureRestAssured filter and what does it do?**

**A:**

Add one line in your `@BeforeClass` or `@BeforeSuite`:

```java
@BeforeClass
public void setUp() {
    RestAssured.baseURI = "https://api.example.com";
    RestAssured.filters(new AllureRestAssured());  // single line
}
```

This filter intercepts every RestAssured request and response and attaches the full details to the Allure test result automatically:
- Full request URL (including query parameters)
- Request headers (including Authorization, Content-Type)
- Request body (JSON payload)
- HTTP response status code
- Response headers
- Response body

Without this filter, you would need to manually call `Allure.addAttachment(...)` after every API call. The filter does it transparently for every call in the test.

You can combine it with other filters:
```java
RestAssured.filters(
    new AllureRestAssured(),          // Allure logging
    new RequestLoggingFilter(),       // also log to console
    new ResponseLoggingFilter()       // also log response to console
);
```

For per-request filtering instead of global:
```java
given()
    .filter(new AllureRestAssured())  // only this request logged to Allure
    .body(payload)
    .post("/api/orders");
```

**Q10: How do you take a screenshot on test failure and attach it to Allure?**

**A:**

Implement a TestNG `ITestListener` and override `onTestFailure`. The key is that the method returning the screenshot bytes must be annotated with `@Attachment`:

```java
package com.example.listeners;

import io.qameta.allure.Attachment;
import org.openqa.selenium.OutputType;
import org.openqa.selenium.TakesScreenshot;
import org.openqa.selenium.WebDriver;
import org.testng.ITestListener;
import org.testng.ITestResult;

public class ScreenshotListener implements ITestListener {

    @Override
    public void onTestFailure(ITestResult result) {
        Object testInstance = result.getInstance();
        if (testInstance instanceof BaseTest) {
            WebDriver driver = ((BaseTest) testInstance).getDriver();
            if (driver != null) {
                takeAndAttachScreenshot(driver);
            }
        }
    }

    @Attachment(value = "Screenshot on failure", type = "image/png", fileExtension = "png")
    private byte[] takeAndAttachScreenshot(WebDriver driver) {
        return ((TakesScreenshot) driver).getScreenshotAs(OutputType.BYTES);
    }
}
```

Register the listener in `testng.xml`:
```xml
<suite name="API Test Suite">
    <listeners>
        <listener class-name="com.example.listeners.ScreenshotListener"/>
    </listeners>
    <test name="Order API Tests">
        <classes>
            <class name="com.example.tests.CreateOrderTest"/>
        </classes>
    </test>
</suite>
```

Important details:
- The return type must be `byte[]` for images — Allure reads these bytes and embeds the image
- The method must be annotated `@Attachment` — not just called with the bytes
- `getDriver()` must be accessible from the listener — expose it as a `public` method in your `BaseTest` class

For API-only tests without WebDriver, attach the response body instead:
```java
@Override
public void onTestFailure(ITestResult result) {
    Object instance = result.getInstance();
    if (instance instanceof BaseApiTest) {
        Response lastResponse = ((BaseApiTest) instance).getLastResponse();
        if (lastResponse != null) {
            attachResponseBody(lastResponse.getBody().asString());
        }
    }
}

@Attachment(value = "Last API Response", type = "application/json", fileExtension = "json")
private String attachResponseBody(String body) {
    return body;
}
```

**Q11: What commands do you use to generate and serve an Allure report?**

**A:**

```bash
# Step 1: Run tests — generates allure-results/ directory with .json data files
mvn test

# Step 2a: Serve an interactive report in the browser (best for local development)
# Opens browser automatically at http://localhost:PORT/
allure serve allure-results/

# Step 2b: Generate a static HTML report (best for CI artifacts, sharing, archiving)
allure generate allure-results/ --clean -o allure-report/
# --clean: delete allure-report/ before generating (prevents stale data)
# -o allure-report/: output directory

# Step 3: Open the generated static report
allure open allure-report/
# Opens allure-report/index.html in the default browser

# Install Allure CLI (if not installed)
npm install -g allure-commandline   # via npm
brew install allure                  # macOS via Homebrew
# Windows: download from https://github.com/allure-framework/allure2/releases

# Check version
allure --version

# Preserve history between runs (for trend graphs)
# Copy allure-report/history/ into allure-results/ before generating
cp -r allure-report/history allure-results/
allure generate allure-results/ --clean -o allure-report/
```

The difference between `allure serve` and `allure generate`:
- `allure serve`: starts a temporary local web server, opens browser, deletes server when you close it. Best for quickly viewing results locally
- `allure generate`: creates a static `index.html` that you can open without a server, share as a zip, or publish as a CI artifact. Required for Jenkins/GitHub Actions

**Q12: How do you integrate Allure with Jenkins?**

**A:**

1. Install the **Allure Jenkins Plugin** from Jenkins Plugin Manager (Manage Jenkins > Plugins > Available)

2. Configure the Allure installation in Jenkins (Manage Jenkins > Global Tool Configuration > Allure Commandline > Add Allure Commandline)

3. Add the Allure post-build step in your Jenkinsfile:

```groovy
pipeline {
    agent any

    stages {
        stage('Checkout') {
            steps {
                git branch: 'main', url: 'https://github.com/your-org/your-repo.git'
            }
        }

        stage('Test') {
            steps {
                sh 'mvn clean test'
            }
        }
    }

    post {
        always {
            // Generate Allure report from allure-results/ directory
            allure([
                includeProperties: false,
                jdk: '',
                results: [[path: 'allure-results']],
                // Optional: specify report path
                report: 'allure-report'
            ])
        }
        failure {
            emailext(
                subject: "Build FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                body: "Tests failed. See Allure report: ${env.BUILD_URL}allure/",
                to: 'qa-team@company.com'
            )
        }
    }
}
```

After each build:
- The Allure Report appears as a **sidebar link** called "Allure Report" on the Jenkins build page
- The plugin automatically maintains **history** between builds — the trend graph shows the last N builds
- You can also access it at `http://jenkins-url/job/your-job/allure/`

The `post { always { } }` block is critical — it ensures the report is generated even when tests fail. If you put `allure()` in the `steps` block, it is skipped if a test stage fails.

**Q13: What is ExtentReports and how is it different from Allure?**

**A:**

ExtentReports is a Java library (from Aventstack) that generates self-contained HTML test reports. Unlike Allure, ExtentReports does not require a separate CLI tool to render the report — the HTML file is generated directly by your Java test code using the ExtentReports API. The resulting HTML file is completely self-contained (CSS, JS, charts all embedded) and can be opened in any browser without a server.

Key differences:

| Feature | Allure | ExtentReports |
|---------|--------|---------------|
| CLI tool required | Yes (allure generate) | No — report generated by Java code |
| External links (JIRA) | Yes (@Issue, @TmsLink) | Manual links only |
| Trend history | Yes (built-in) | No (requires external tooling) |
| Severity classification | Yes (5 levels) | Yes (via categories) |
| Thread safety | Yes | Requires ThreadLocal setup |
| RestAssured integration | Yes (AllureRestAssured filter) | Manual (via logStep) |
| Report quality | Very rich, interactive | Rich, simpler than Allure |
| Best for | Large suites, CI with Jenkins | Quick setup, smaller teams |

Maven dependency:
```xml
<dependency>
    <groupId>com.aventstack</groupId>
    <artifactId>extentreports</artifactId>
    <version>5.1.1</version>
    <scope>test</scope>
</dependency>
```

**Q14: Show a complete ExtentReports setup with a TestNG listener.**

**A:**

Three classes are needed: `ExtentManager` (singleton), `ExtentTestListener` (TestNG listener), and usage in your test class.

```java
// ExtentManager.java — singleton, initialises the report once
package com.example.reporting;

import com.aventstack.extentreports.ExtentReports;
import com.aventstack.extentreports.reporter.ExtentSparkReporter;
import com.aventstack.extentreports.reporter.configuration.Theme;

public class ExtentManager {

    private static ExtentReports extent;

    public static synchronized ExtentReports getInstance() {
        if (extent == null) {
            ExtentSparkReporter spark = new ExtentSparkReporter("test-output/ExtentReport.html");
            spark.config().setTheme(Theme.DARK);
            spark.config().setDocumentTitle("Test Execution Report");
            spark.config().setReportName("Order API Automation Suite");
            spark.config().setTimeStampFormat("dd-MMM-yyyy HH:mm:ss");

            extent = new ExtentReports();
            extent.attachReporter(spark);
            extent.setSystemInfo("Environment", System.getProperty("env", "staging"));
            extent.setSystemInfo("Author", "QA Team");
            extent.setSystemInfo("Java", System.getProperty("java.version"));
            extent.setSystemInfo("OS", System.getProperty("os.name"));
        }
        return extent;
    }
}
```

```java
// ExtentTestListener.java — TestNG listener
package com.example.listeners;

import com.aventstack.extentreports.ExtentReports;
import com.aventstack.extentreports.ExtentTest;
import com.aventstack.extentreports.MediaEntityBuilder;
import com.aventstack.extentreports.Status;
import com.example.reporting.ExtentManager;
import org.openqa.selenium.OutputType;
import org.openqa.selenium.TakesScreenshot;
import org.openqa.selenium.WebDriver;
import org.testng.ITestContext;
import org.testng.ITestListener;
import org.testng.ITestResult;

public class ExtentTestListener implements ITestListener {

    // ThreadLocal ensures each parallel test thread has its own ExtentTest node
    private static final ThreadLocal<ExtentTest> testThread = new ThreadLocal<>();
    private static final ExtentReports extent = ExtentManager.getInstance();

    public static ExtentTest getTest() {
        return testThread.get();
    }

    @Override
    public void onTestStart(ITestResult result) {
        ExtentTest test = extent.createTest(
            result.getMethod().getMethodName(),
            result.getMethod().getDescription()
        );
        testThread.set(test);
        test.log(Status.INFO, "Test started on thread: " + Thread.currentThread().getId());
    }

    @Override
    public void onTestSuccess(ITestResult result) {
        testThread.get().log(Status.PASS, "Test PASSED — duration: " +
            (result.getEndMillis() - result.getStartMillis()) + "ms");
    }

    @Override
    public void onTestFailure(ITestResult result) {
        ExtentTest test = testThread.get();
        test.log(Status.FAIL, "Test FAILED");
        test.log(Status.FAIL, result.getThrowable());  // logs full stack trace

        // Attach screenshot if WebDriver available
        Object instance = result.getInstance();
        if (instance instanceof BaseTest) {
            WebDriver driver = ((BaseTest) instance).getDriver();
            if (driver != null) {
                String base64 = ((TakesScreenshot) driver)
                    .getScreenshotAs(OutputType.BASE64);
                test.fail("Browser at failure point:",
                    MediaEntityBuilder.createScreenCaptureFromBase64String(base64).build());
            }
        }
    }

    @Override
    public void onTestSkipped(ITestResult result) {
        ExtentTest test = testThread.get();
        test.log(Status.SKIP, "Test SKIPPED");
        if (result.getThrowable() != null) {
            test.log(Status.SKIP, result.getThrowable());
        }
    }

    // Call from test code to log individual steps
    public static void logStep(Status status, String message) {
        ExtentTest test = testThread.get();
        if (test != null) {
            test.log(status, message);
        }
    }

    @Override
    public void onFinish(ITestContext context) {
        // CRITICAL: flush() writes the HTML to disk — without this, no report is generated
        extent.flush();
    }
}
```

```java
// Using logStep in your test
@Test
public void createOrder_returns201() {
    ExtentTestListener.logStep(Status.INFO, "Building order payload");
    String payload = "{\"customerId\":\"CUST-42\",\"amount\":99.99}";

    ExtentTestListener.logStep(Status.INFO, "Sending POST /api/orders");
    Response response = given().body(payload).post("/api/orders");

    assertEquals(response.getStatusCode(), 201);
    ExtentTestListener.logStep(Status.PASS, "Status 201 confirmed");

    String orderId = response.jsonPath().getString("orderId");
    assertNotNull(orderId);
    ExtentTestListener.logStep(Status.PASS, "orderId present: " + orderId);
}
```

Register in `testng.xml`:
```xml

    <listeners>
        <listener class-name="com.example.listeners.ExtentTestListener"/>
    </listeners>
</suite>
```

## SECTION 4 — PLAYWRIGHT HTML REPORT

---

**Q15: What is Playwright's built-in HTML report and what does it contain?**

**A:**

Playwright's HTML reporter is included with `@playwright/test` — no additional packages required. Running `npx playwright test` with the HTML reporter generates a `playwright-report/` directory containing `index.html` and all assets.

The report contains:

| Section | Content |
|---------|---------|
| Summary bar | Total tests, passed, failed, skipped, duration, pass rate |
| Test list | Each test with PASSED/FAILED/SKIPPED status and duration |
| Error detail | The specific assertion that failed, expected vs received values |
| Stack trace | File path and line number where the failure occurred |
| Screenshot | Full-page screenshot captured automatically at failure |
| Video | Replay of the full test browser session (if configured) |
| Network log | All fetch/XHR requests with URL, status code, timing |
| Trace viewer | Step-by-step DOM snapshots — see the page state at every action |

The trace viewer is the most powerful debugging tool. It shows:
- A timeline bar with every action (click, fill, navigate, expect)
- A DOM snapshot before and after each action (you can see exactly what the page looked like)
- Console messages at that moment
- Network requests made during that step

This makes Playwright's HTML report the best debugging tool of any test reporter — you can replay a test failure step by step like a video with full DOM access.

**Q16: How do you configure the Playwright reporter in playwright.config.ts?**

**A:**

```typescript
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  timeout: 30000,       // 30 seconds per test
  retries: 1,           // retry failing tests once

  reporter: [
    // Primary: rich HTML report with trace, screenshots, video
    ['html', {
      outputFolder: 'playwright-report',  // output directory
      open: 'on-failure'                  // 'always' | 'on-failure' | 'never'
    }],

    // Terminal output while tests run
    ['list'],

    // JUnit XML for CI systems (Jenkins test results parser, GitHub Actions)
    ['junit', { outputFile: 'test-results/junit.xml' }],

    // JSON output for custom processing
    ['json', { outputFile: 'test-results/results.json' }],
  ],

  use: {
    baseURL: 'https://api.example.com',

    // Screenshot options: 'on' | 'off' | 'only-on-failure'
    screenshot: 'only-on-failure',

    // Video options: 'on' | 'off' | 'retain-on-failure' | 'on-first-retry'
    video: 'retain-on-failure',

    // Trace options: 'on' | 'off' | 'retain-on-failure' | 'on-first-retry'
    trace: 'on-first-retry',
  },

  // Parallel execution configuration
  workers: process.env.CI ? 2 : undefined,

  projects: [
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
    },
    {
      name: 'firefox',
      use: { ...devices['Desktop Firefox'] },
    },
  ],
});
```

Explanation of `on-first-retry` for trace: this means the trace is captured only when a test is being retried (i.e., it failed once). This keeps the trace file storage small for passing tests while ensuring you always have a trace when a test is flaky or failing.

**Q17: How do you configure screenshot, video, and trace capture on failure?**

**A:**

```typescript
// playwright.config.ts — configure evidence capture per environment
export default defineConfig({
  use: {
    // SCREENSHOT
    // 'on'             — screenshot after every action (large storage)
    // 'off'            — never take screenshots
    // 'only-on-failure'— screenshot only when test fails (recommended)
    screenshot: 'only-on-failure',

    // VIDEO
    // 'on'             — record video for every test (large storage)
    // 'off'            — no video
    // 'retain-on-failure' — record video, delete if test passes (recommended)
    // 'on-first-retry' — record only when retrying
    video: 'retain-on-failure',

    // TRACE
    // 'on'             — trace for every test (DOM snapshots, network, console)
    // 'off'            — no trace
    // 'retain-on-failure' — trace but delete if test passes
    // 'on-first-retry' — trace only when retrying (recommended for CI)
    trace: 'on-first-retry',
  },
});
```

Screenshots and videos are stored in `test-results/` directory by default. Each test gets its own subdirectory named after the test. After running, you can view them:

```bash
# Open the HTML report (includes screenshots, video, trace links)
npx playwright show-report

# Open a specific trace file directly
npx playwright show-trace test-results/order-api-create-order/trace.zip
```

Inside a test you can also take a manual screenshot:
```typescript
test('verify order appears in list', async ({ page }) => {
    // manual screenshot at a specific point
    await page.screenshot({ path: 'before-click.png', fullPage: true });

    await page.screenshot({ path: 'after-click.png', fullPage: true });

```

**Q18: What does `npx playwright show-report` do?**

**A:**

`npx playwright show-report` starts a local HTTP server and opens the Playwright HTML report in your default browser. It reads the `playwright-report/` directory (or the folder specified in config under `reporter: [['html', { outputFolder: ... }]]`).

```bash
# Run tests and then view report
npx playwright test
npx playwright show-report

# Specify a different report directory
npx playwright show-report path/to/my-report/

# Open a specific port
npx playwright show-report --host localhost --port 9323
```

The server remains running until you press Ctrl+C. This is important because the HTML report references local file paths for screenshots and trace files — the server serves these correctly. If you just open `index.html` directly in the browser as a file, the screenshots may not load due to browser security restrictions on local file access.

For CI environments you typically use one of:
1. Upload `playwright-report/` as a CI artifact (GitHub Actions, Jenkins)
2. Deploy it to a static hosting service (GitHub Pages, Netlify, S3)
3. Use `allure-playwright` for the richer Allure dashboard

## SECTION 6 — REPORT QUALITY AND BEST PRACTICES

---

**Q20: What makes a test report genuinely useful? What should it contain?**

**A:**

A genuinely useful test report answers different questions for different audiences — without requiring those audiences to read code.

For a Release Manager (5-second view):
- Total tests / passed / failed / skipped
- Overall pass rate (e.g. 89.1%)
- Total duration
- Trend arrow: better or worse than last build?

For a Product Owner (1-minute view):
- Which features/epics have failures?
- Are the failing tests BLOCKER severity or minor?
- Is this a new failure (regression) or a known issue?

For a Developer investigating a failure (5-minute view):
- Which specific test failed?
- At which step did it fail?
- What was the request sent to the API?
- What was the response received?
- What does the page/app look like at the point of failure (screenshot)?
- What is the error message and stack trace?

A complete test report contains:
1. Executive summary (pass rate, counts, duration, trend)
2. Failure list grouped by feature area and severity
3. Per-test detail: step breakdown, assertion failure message, screenshots/videos/API logs
4. Test metadata: environment name, build number, branch, run by
5. External links: JIRA issue, test case in Xray/Zephyr
6. History/trend: comparison with previous N builds
7. Categorised failures: new regression vs known vs environment vs flaky

The reporting hierarchy should be navigable without touching code:
```
Suite level → Feature level → Test level → Step level → Evidence
```

**Q21: How do you show test trends in Allure and Jenkins?**

**A:**

Trend data requires keeping results history between runs. Allure stores historical data in a `history/` folder.

In Jenkins with the Allure Jenkins Plugin:
```groovy
post {
    always {
        allure([results: [[path: 'allure-results']]])
        // The plugin automatically copies history from the previous build
        // and makes it available for the next run
    }
}
```

Jenkins maintains the history automatically — the trend graph in the sidebar shows the last 30 builds by default.

For manual history management (GitHub Actions, local):
```bash
# Before generating this run's report, copy history from the previous report
mkdir -p allure-results/history
cp -r allure-report/history/. allure-results/history/

# Generate with history
allure generate allure-results/ --clean -o allure-report/
```

In GitHub Actions:
```yaml
- name: Load previous Allure history

  run: mvn test

- name: Upload Allure history for next run
  uses: actions/upload-artifact@v4
  with:
    name: allure-history
    path: allure-report/history
    retention-days: 90
```

The Allure trend graph shows pass rate, fail rate, and duration per build. The Retries Trend shows how often tests are retried. The Duration Trend shows whether tests are getting slower over time.

**Q22: How do you flag flaky tests in Playwright and Allure reports?**

**A:**

A flaky test is a test that produces inconsistent results — passes sometimes and fails sometimes without any code change. Causes include timing issues, shared test data, race conditions, or genuine environmental instability.

In Playwright — configure retries, and Playwright marks flaky tests automatically:
```typescript
// playwright.config.ts
export default defineConfig({
  retries: 2,  // retry failing tests up to 2 times
});
```

If a test fails on the first run but passes on a retry, Playwright marks it as "Flaky" in the HTML report and the JUnit XML with a special badge. The test is considered "passed" (since it eventually passed) but the flakiness is visible to the team.

In Allure with Java — the same logic applies. Configure retries via TestNG:
```xml
<!-- testng.xml -->
<suite retry-analyzer="com.example.listeners.RetryAnalyzer">
```

```java
public class RetryAnalyzer implements IRetryAnalyzer {
    private int count = 0;
    private static final int MAX_RETRY = 2;

    @Override
    public boolean retry(ITestResult result) {
        if (!result.isSuccess() && count < MAX_RETRY) {
            count++;
            return true;  // retry the test
        }
        return false;
    }
}
```

Allure automatically detects when a test passed after a retry and marks it with a "Flaky" badge in the Categories tab. In the Allure dashboard:
- Failed tests that were retried and eventually passed → **Flaky** category
- Tests that failed on all retries → **Product defects** or **Test defects** category

Best practice: **never suppress flaky tests by retrying silently**. Make flakiness visible in the report. Track which tests are consistently flaky — they represent real risk. A test that needs 3 retries to pass is not actually passing; it is masking a problem.

## SECTION 7 — INTERVIEW Q&A

---

**Q23: Why do test reports matter in a QA process?**

**A:** Automated tests run hundreds of assertions across many files. The raw output — stack traces, console logs, JUnit XML — is meaningful to the developer who wrote the test, but completely opaque to everyone else who needs to make decisions based on it. A Product Owner needs to know which features are healthy. An Engineering Manager needs to know whether the release is safe to ship. A Release Manager needs to know whether the build is green enough to promote to production.

Good reports translate technical pass/fail data into business-readable information. They also serve as a permanent historical record — you can demonstrate over time that quality is improving (or flag that it is degrading) with trend graphs. Without good reporting, your automation runs but nobody outside QA can make informed decisions from it.

---

**Q24: What is the difference between Allure, ExtentReports, and Playwright HTML report?**

**A:** All three generate HTML test reports but differ significantly in capability and integration. **Allure** is the most feature-rich — hierarchical organisation (epic/feature/story), severity levels, external system links, automatic request/response logging for RestAssured, built-in trend history, and a rich interactive dashboard. Requires a separate CLI tool. Best for Java/TestNG suites integrated with JIRA. **ExtentReports** is a lightweight Java library — no external CLI, report is generated by calling `extent.flush()` in `@AfterSuite`. Simpler setup, no built-in trends or external links. Good for teams that want a quick rich HTML report without infrastructure overhead. **Playwright HTML** is built into Playwright — zero extra setup. It is uniquely powerful for debugging because it includes video replay, trace viewer (DOM snapshots at every action), and network request logs. Specific to Playwright, does not support Java/TestNG.

---

**Q25: How do you attach the API request and response to every test result automatically?**

**A:** For RestAssured, add `RestAssured.filters(new AllureRestAssured())` in `@BeforeClass` or `@BeforeSuite`. This single line makes every RestAssured request and response appear in the Allure report as a named attachment with full headers and body — no additional code per test is needed. For Playwright, the HTML reporter automatically includes all network requests in the test detail view. For ExtentReports, log the request and response manually in a helper method that calls `ExtentTestListener.logStep()`, or use a RestAssured filter that logs to ExtentReports. The AllureRestAssured filter is the most elegant solution — one line of setup covers the entire test suite.

---

**Q26: How do you prevent test reports from missing failures when tests run in parallel?**

**A:** The main risk with parallel test execution and reporting is thread safety — if two test threads both try to write to the same report object simultaneously, data gets corrupted or mixed between tests. For **Allure**, this is handled automatically because each test gets its own JSON result file written to disk — there is no shared in-memory state per test. For **ExtentReports**, use `ThreadLocal<ExtentTest>` — each thread stores its own `ExtentTest` node, so there is no sharing. The `ExtentManager` (the `ExtentReports` instance itself) can be shared since writing to it is safe, but the `ExtentTest` per test must be thread-local. The `extent.flush()` call in `onFinish` is called once by the last thread to finish, which writes all thread-local data to the report.

**Q27: What is a test report category and how do you configure categories in Allure?**

**A:** Allure Categories let you group failures by their cause — distinguishing product bugs from test bugs from infrastructure issues. You define categories in a `categories.json` file placed in `allure-results/`:

```json
[
  {
    "name": "Product defects",
    "matchedStatuses": ["failed"],
    "messageRegex": ".*AssertionError.*"
  },
  {
    "name": "Test defects",
    "matchedStatuses": ["broken"],
    "messageRegex": ".*NullPointerException.*|.*NoSuchElementException.*"
  },
  {
    "name": "Environment issues",
    "matchedStatuses": ["failed", "broken"],
    "messageRegex": ".*Connection refused.*|.*timeout.*"
  }
]
```

A test is "Failed" when an assertion fails (product bug). It is "Broken" when an unexpected exception prevents the test from completing (test bug or environment issue). The Categories tab in Allure shows counts per category, making it easy for a lead to see "10 product defects, 3 broken tests, 2 environment failures" rather than just "15 failures."

**Q28: How do you publish a Playwright report to GitHub Actions for download?**

**A:**
```yaml
# .github/workflows/e2e.yml
name: E2E Tests

on: [push, pull_request]

jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright browsers
        run: npx playwright install --with-deps

      - name: Run Playwright tests
        run: npx playwright test

      - name: Upload Playwright report
        if: always()   # upload even when tests fail
        uses: actions/upload-artifact@v4
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30

      - name: Upload test results (JUnit XML)
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: test-results/junit.xml
```

The `if: always()` is essential — without it, the upload step is skipped if any previous step fails (including the test step when tests fail), meaning you never get the report for the runs that matter most. After the workflow runs, the report is downloadable from the GitHub Actions run page under "Artifacts."

**Q29: What is the Playwright Trace Viewer and when do you use it?**

**A:** The Playwright Trace Viewer is a built-in debugging tool that records a complete "time machine" of a test — every action, DOM snapshot before and after, console messages, and network requests. It is the most powerful test debugging tool in any framework currently available.

Configure it in `playwright.config.ts`:
```typescript
use: {
  trace: 'on-first-retry',  // capture trace only on retry (recommended for CI)
  // trace: 'on',           // always capture (large storage, good for debugging locally)
}
```

View the trace:
```bash
# Via the HTML report (click the "Trace" link on any failed test)
npx playwright show-report

# Open a specific trace file directly
npx playwright show-trace test-results/test-name/trace.zip
```

The Trace Viewer shows:
- A timeline at the top with every action
- Click any action to jump to that moment in time
- Left panel: DOM snapshot at that moment (the exact HTML the test saw)
- Right panel: console logs, network requests, source location
- Before/after snapshots: see the DOM before and after each click/fill

Use it when a test fails intermittently and you cannot reproduce it locally — the trace gives you exact evidence of what the application showed at every step, making it possible to diagnose even race conditions and timing issues.

**Q30: How do you flag a test as a known failure (expected to fail) in Allure without marking it as a failure?**

**A:** Use `@Flaky` for tests known to be intermittent, or handle expected failures in TestNG with `expectedExceptions` and `expectedExceptionsMessageRegExp`. For tests that are genuinely expected to fail (e.g. a known bug not yet fixed), the options are:

```java
// Option 1: TestNG expectedExceptions — test PASSES if the exception is thrown
@Test(expectedExceptions = RuntimeException.class,
      expectedExceptionsMessageRegExp = ".*Service unavailable.*")
public void knownBug_OMS1234_serviceReturns503() {
    // When OMS-1234 is fixed, this test will FAIL (because exception no longer thrown)
    // That failure is the signal the bug was fixed
    orderService.createOrder(payload);
}

// Option 2: Allure @Flaky annotation — marks test as known flaky
// Add to the test method — Allure shows it with a flaky badge regardless of result
// Use with: allure-junit5 or allure-testng supporting @Flaky

// Option 3: TestNG @Test(enabled=false) — skip entirely with a comment
@Test(enabled = false)
// Disabled: OMS-1234 — known defect, revisit after sprint 42
public void knownBrokenTest() { ... }
```

Best practice: do not use `enabled=false` to hide broken tests. Use `expectedExceptions` to document that you know the current behaviour is wrong, so the test automatically becomes a regression detector when the bug is fixed.
